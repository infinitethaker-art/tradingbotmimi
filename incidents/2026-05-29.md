## Root-cause findings

### CONFIRMED — not a DB wipe
Every restart logs `Mounting volume on: .../vol_2ow6a88qry8din8w` — **same volume ID**. The SQLite DB on `/app/db` persists across restarts. The "ephemeral storage" theory is rejected.

### CONFIRMED — alert storm is a design defect
On reconcile HALT, `main()` exits; `railway.toml restartPolicyType=ALWAYS` and the `while True` outer loop in `scheduler/main.py:239` relaunch it every ~36s. No backoff, no alert dedup. A halted bot should stay down quietly and alert ONCE.

### CONFIRMED — reconciler flags bracket legs as orphans (`reconciler.py:119-132`)
A bracket entry, once filled, leaves TP + SL child orders resting at the broker. Those child IDs are never written to `order_events` (acknowledged at `trade_logger.py:259`, "bracket child order — expected"). The orphan check compares ALL open broker orders to the DB → child legs always look like orphans. **Any restart while a bracketed position is open re-HALTs.**

### CONFIRMED — phantom-position race (`loop_b.py:177-205`, `position_tracker.py:66-114`)
On a buy fill, `on_fill_event` updates `order_events` and sends the `FILL` Telegram alert FIRST; the `positions` row is INSERTed afterward in `on_entry_fill`, non-atomically. A crash/restart/missed-WS-event between the two yields a broker position with no DB row (= phantom) even though a FILL alert was sent.

### CONFIRMED (via live DB dump, 2026-05-29) — root cause is yesterday's DOUBLE-ENTRY
The live `order_events`/`positions` dump settled it:
- 05-28 `order_events` has **two AAPL buys** (`8fb81a39` 14:00, `cbf88732` 14:15) and **two NVDA buys** (`d25a9690` 13:45, `68fadda5` 14:00), all filled — but `positions` has only **one** AAPL (qty 2) and **one** NVDA (qty 3).
- So the bot double-entered AAPL and NVDA on consecutive bars. Both copies filled at the broker, but `PositionTracker.on_entry_fill`'s "already has an open position" guard (position_tracker.py:78) recorded only one position per symbol — the duplicate fill's shares were bought at the broker but dropped from DB tracking.
- End-of-day time-exit sold only the DB-tracked qty, leaving the **untracked duplicate shares open at the broker** (NVDA, AAPL) plus their **bracket legs resting as orphans** (`aeedccdf`, `25e70c74` — confirmed NOT present anywhere in `order_events`).
- Next morning's reconcile correctly saw broker positions + orders with no open DB record → phantom + orphan → HALT. The reconciler did its job; the bug was upstream.

**The double-entry trigger is already fixed in deployed code:** `loop_a.py:161` now rejects re-entry with `SYMBOL_ALREADY_OPEN` when `position_tracker.has_open(symbol)`. Yesterday ran older code without this guard; today's (05-29) orders show NO duplicates (1 AAPL buy, 1 TSLA buy) — consistent with the fix.

**Phantom-INSERT race (#3) is RULED OUT as the cause:** today's AAPL + TSLA recorded perfectly (buy+sell orders filled, positions opened+closed). The fill→DB path is healthy. #3 remains only a minor latent nit, not this incident.

## Recommended fixes (all touch protected dirs — need explicit approval)
1. `reconciler.py` Check 4: exclude bracket child/leg orders from orphan detection (match by `legs`/parent, or only flag standalone non-bracket orders).
2. HALT should be sticky + quiet: write a halt flag, alert ONCE, stop the restart-thrash (touches `scheduler/main.py` + reconcile flow).
3. Make entry persistence atomic/ordered: write the `positions` row BEFORE sending the FILL alert, or in the same transaction as the `order_events` update.
4. (Design) Give reconciler a safe ADOPT-or-CLOSE path for unrecognized broker state instead of halt-forever.
